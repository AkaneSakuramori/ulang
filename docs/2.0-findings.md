# 2.0 Real-World Findings Log

A record of genuine limitations discovered by building real software in Ulang, and the
improvements made in response. This log is a defining outcome of the 2.0 milestone: every
entry is a shortcoming that only surfaced through practical use, not speculation.

## Summary

Eleven reference projects (`projects/`) were built across diverse domains — expression
evaluation, text processing, data modelling, cellular automata, error-handling pipelines,
formatting, an in-memory database, numerics, a Lisp interpreter, a file-based report
generator, and graph search. They surfaced the following, all fixed unless noted:

| # | Area | Exposed by | Outcome |
|---|---|---|---|
| F1 | `list.sort` ignored its comparator | wordstats | Fixed: optional comparator, both engines |
| F2 | match arms take blocks/exprs, not braces | calc | By design; documented |
| F3 | missing `str.substring`/`index_of`/`repeat` | calc, rpn, jsonfmt, table | Fixed: added, both engines |
| F4 | recursion capped at 2500 frames | deep-recursion probe | Fixed: raised to 5000 |
| F5 | self-hosted driver mis-decoded on Windows | CI matrix | Fixed: force UTF-8 |
| F6 | `math` missing trig/log/min/max/round | stats | Fixed: added, both engines |
| F7 | self-hosted lexer mis-scanned nested strings in `${}` | lisp | Fixed: track interpolation depth |
| F8 | `from` is a reserved keyword | graph | By design; documented |
| F9 | no portable temp path; `/tmp` hardcoded | report (Windows CI) | Fixed: added `platform.tmpdir()` |

After the project-driven fixes above (the last of which, F9, was a cross-platform I/O gap
caught by the Windows CI matrix), broad edge-case probing surfaced no further defects — the
signal that the language is stable for real-world development.

### F1 — `list.sort` ignored its comparator
- **Exposed by:** `wordstats` (sorting `(word, count)` pairs by frequency).
- **Problem:** `list.sort()` was implemented as a bare `sorted(obj)`, ignoring any argument
  and throwing `TypeError` when the elements were structs (no natural ordering).
- **Fix:** `list.sort(cmp)` now accepts an optional comparator `fn(a, b) -> int` (negative /
  zero / positive) or a boolean, in addition to the no-argument natural sort. Implemented
  once in `src/builtins_mod.py` (`_sort`), shared by the interpreter and the VM.
  Backward-compatible. (CHANGELOG: "toward 2.0".)

### F2 — match arms don't take brace blocks (ergonomics, documented, no change)
- **Exposed by:** `calc` (wanting a multi-statement `match` arm).
- **Finding:** match arms are a single expression or an indented block (per the grammar);
  the `{ ... }` brace form is only for lambda bodies. This is a deliberate grammar choice,
  not a bug. Resolved by writing idiomatic arms; recorded so the constraint is explicit.

### F3 — missing common string methods (`substring`, `index_of`, `repeat`)
- **Exposed by:** `calc`, `rpn`, `jsonfmt` all hand-rolled substring/number-formatting
  helpers; `table` needed column padding.
- **Problem:** the string type had `split`/`trim`/`replace`/`contains`/`starts_with`/
  `ends_with`/`chars`/`upper`/`lower` but no way to take a slice, find a substring index, or
  repeat a string — forcing every text-processing program to reimplement them by hand.
- **Fix:** added `str.substring(start[, end])` (bounds-clamped), `str.index_of(sub)`
  (returns `-1` when absent), and `str.repeat(n)`, in `src/builtins_mod.py`, shared by the
  interpreter and VM. Backward-compatible. `table` uses them directly for padding.

### F4 — recursion depth capped at 2500 frames, far below the available stack
- **Exposed by:** probing deep recursion while assessing readiness for recursive-descent
  and deep-data-structure workloads (`sum_to(5000)`, and generally any algorithm recursing
  more than ~2500 deep).
- **Problem:** the interpreter and VM hard-capped recursion at 2500 frames, even though the
  runtime already executes on a 256 MB native thread stack (`src/bigstack.py`) that safely
  supports roughly 6000 frames. Legitimate deep recursion panicked well before the real
  limit.
- **Fix:** raised the cap to 5000 in both engines (verified safe — 4900-deep recursion runs
  without a native crash, comfortably within the ~6000-frame budget of the granted stack).
  Big-integer arithmetic was confirmed correct and arbitrary-precision during the same
  probe (`fact(25)` is exact), so no change was needed there.



### F5 — self-hosted compiler driver mis-decoded stage output on Windows
- **Exposed by:** running the reference projects' self-hosted-compilation check on the
  Windows CI matrix (all projects failed there while passing on Linux/macOS).
- **Problem:** the integrated compiler driver (`src/selfhost_driver.py`) captured each
  stage subprocess's output with `text=True` but no explicit encoding, so on Windows
  (non-UTF-8 default codepage) the canonical S-expression tree piped between stages could
  be mis-decoded, breaking compilation.
- **Fix:** force `encoding="utf-8"` on subprocess capture and `newline=""` on intermediate
  file writes, so the pipeline is byte-exact on every platform. Surfaced stage stderr in
  `tests/test_projects.py` for future diagnosis.

### F6 — `math` module missing common functions
- **Exposed by:** `stats` (needed `sqrt`, `min`, `max`); general numeric probing showed
  trigonometric and logarithmic functions absent.
- **Problem:** the `math` module had only `sqrt`, `pow`, `floor`, `ceil`, `abs`, `pi`, `e` —
  no `sin`/`cos`/`tan`, `log`/`exp`, `min`/`max`, or `round`, which numeric code routinely
  needs.
- **Fix:** added `sin`, `cos`, `tan`, `log` (natural, or `log(x, base)`), `exp`, `min`,
  `max`, and `round` (round-half-up) to `src/stdlib.py`. Available on both engines.

### F7 — self-hosted lexer mis-scanned nested strings inside `${...}` interpolation
- **Exposed by:** `lisp` (source contains `"... ${show(run("(square 9)", env))}"` — a string
  literal nested inside an interpolation expression). The reference compiler handled it, but
  `ulang selfhost` reported false `undefined name` diagnostics for identifiers *inside* the
  nested string literal.
- **Problem:** the self-hosted lexer's string scanner stopped at the first `"` and did not
  track `${ ... }` interpolation, so it resumed lexing the contents of an inner string as
  code. The reference lexer tracks interpolation brace depth.
- **Fix:** the self-hosted lexer (in `parser.ul`, and the standalone `lexer.ul`) now skips a
  `${ ... }` interpolation by tracking brace depth and recursively skipping nested string
  literals, matching the reference. Verified against the lexer/parser/checker conformance
  suites and the bootstrap (all still equivalent to the reference), and `lisp` now
  self-hosts. This narrows the documented interpolation boundary.

### F8 — `from` is a reserved keyword (documented, no change)
- **Exposed by:** `graph` (naming an `add_edge` parameter `from`).
- **Finding:** `from` is a keyword (used by `import ... from` and `extern fn ... from`), so
  it cannot be an identifier. This is correct language design; the fix is to choose another
  name (`src`). Recorded so the reserved-word set is explicit for future authors. The full
  reserved set lives in `src/lexer.py`.

### F9 — no portable temp path; file-writing programs hardcoded `/tmp`
- **Exposed by:** `report` (wrote to `/tmp/ulang_report.txt`), which failed on the Windows
  CI matrix where `/tmp` does not exist.
- **Problem:** the standard library had no portable temporary-directory accessor, so a
  program doing file I/O had to hardcode a platform-specific path.
- **Fix:** added `platform.tmpdir()` (returns the OS temp directory) to `src/stdlib.py`.
  `report` was changed to write a relative filename, and `tests/test_projects.py` now runs
  each project in a throwaway working directory so file-writing projects are portable and
  never litter the repository.
